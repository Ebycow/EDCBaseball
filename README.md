**実況：** リレー！野球とリレーが！

**解説：** そうですると、EPG上でこういう構造が生まれる。

```

NHK BS (SID=101): 20:00〜20:02 「プロ野球　巨人×阪神」  ← リレー起点のスタブ（2分）
```

**実況：** 同じタイトルが！三つ！！20時ちょうどに始まる2分間のイベントが出てきた！！これは何ですか佐藤さん！！

**解説：** これがスタブですよ。単体では何も録画できない。EDCBのイベントリレー機能が「どのチャンネルに追従するか」を判断するためだけに存在する、いわば**起爆装置**みたいなものです。

**実況：** 起爆装置！！なんという危険な響き！！このスタブが検索結果に混入するとどうなりますか！！

**解説：** 同一試合が2件出てきたり、TTRecに意味のない2分間の予約が追加されたりする。

**実況：** 2分間の予約！！2分間録れても試合は録れない！！それどころか混乱を生む！！これをどう処理するか！！

**解説：** 判定はシンプルです。`nhk: True` のチャンネルで、`duration_sec < 300`（5分未満）のイベントは問答無用でスキップする。


## ！！TTRec連携！！「予約という名の試合前のスタメン登録」

---

**実況：** 最後に！`--ttrec` フラグによるTTRec連携！！ファイルは [edcbaseball/ttrec.py](edcbaseball/ttrec.py)！！

**解説：** 検索結果をそのまま録画予約に送り込める機能です。

**実況：** 見つけたその場で予約！シームレス！電光石火！！

**解説：** 予約を送る前に重複チェックをしますね。2種類のチェックがある。

```python
def _same_event(lhs: dict, rhs: dict) -> bool:
    return (
        lhs.get('onid') == rhs.get('networkId')
        and lhs.get('tsid') == rhs.get('transportStreamId')
        and lhs.get('sid') == rhs.get('serviceId')
        and lhs.get('eid') == rhs.get('eventId')
    )

def _same_timeslot(lhs: dict, rhs: dict) -> bool:
    start = lhs['starttime'].strftime('%Y-%m-%dT%H:%M:%S')
    return (
        rhs.get('networkId') == lhs.get('onid')
        and rhs.get('transportStreamId') == lhs.get('tsid')
        and rhs.get('serviceId') == lhs.get('sid')
        and rhs.get('startTime') == start
        and int(rhs.get('duration') or 0) == lhs.get('duration_sec')
    )
```

**実況：** 第一の重複チェックは `onid + tsid + sid + eid`！イベントIDの完全一致！！第二は時刻と長さ！！なぜ二種類のチェックが必要なんですか！！

**解説：** イベントIDはEPGが更新されると変わることがある。でも同じ試合なら放送局・開始時刻・長さは変わらない。どちらかで一致すれば「もう予約済み」として扱う。

**実況：** 堅牢な二重確認！一つの扉が閉まっていても別の扉で確認する！セキュリティゲートの多重チェックのような安全性！！

**解説：** NHKのイベントリレー処理との関係もあってですね、予約ターゲットは `_select_ttrec_targets` で1件に統合した上でTTRecに送ります。重複と追従省略を別々にカウントして表示するので、「何件追加されたか」が一目瞭然です。

**実況：** 透明性！ユーザーへの説明責任！予約の結果を隠さず全部見せる！これこそがエンジニアリングの誠実さですよ！！
